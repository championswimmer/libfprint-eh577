## Summary

Two guided finger-interaction tests were run:

1. interrupt polling with two touch/hold/remove cycles
2. full post-init replay while holding a finger on the sensor

## Interrupt result

The guided interrupt test produced **no interrupt payloads** on `0x83` or `0x84` during:

- finger absent
- finger touch
- finger hold
- finger removal

Within the tested window, interrupts appear unnecessary for the currently observed bulk-capture path.

## Repeat-path finger hold result

A repeat-path run with the finger held on the sensor still returned the same all-zero payload as idle:

- `64 14 ec`
- length `5356`
- nonzero `0`
- SHA-256 `bd2386946403b82cd740767b15507502a27a7d5b4c86440a5af0966c71ad8a38`

## Post-init finger hold result

A post-init run with the finger held on the sensor produced the first meaningful non-zero large payload:

- packet `64 14 ec`
- length `5356`
- nonzero `1305`
- unique byte values `80`
- SHA-256 `51e6dcd08a54f9d1aeed5269e362e4720c2c3835994c586bdac299df569c95c3`

This payload differs from the idle payload in `1305` byte positions.

The payload was converted to PGM using:

- `tools/eh577_dump_to_pgm.py`

and an ASCII preview was saved in:

- `logs/2026-06-03-eh577-postinit-fingerhold-01-ascii.txt`

## Structural observations on the non-zero payload

Assuming EH575 image geometry (`103 x 52`):

- the payload size exactly matches one frame of `103 * 52`
- non-zero bounding box spans:
  - x = `0..69`
  - y = `0..51`
- columns `70..102` are all zero in this capture

This strongly suggests the payload is structured image-like data rather than random noise.

## Small reply changes during post-init finger hold

Compared with the earlier idle post-init dump, several status-like replies changed:

- `60 01 fc`: `... 05 01` -> `... 00 01`
- `60 40 fc`: `... 80 01` -> `... 00 01`
- `60 2d 02`: `... c7 01` -> `... 47 01`
- `62 67 03`: `... ff 00` -> `... 22 08`

This reinforces that EH577 reply bytes are stateful and can reflect finger/contact/capture state.

## Current interpretation

The strongest current working model is:

1. EH577 is an EH575-family bulk image device.
2. The post-init path is important for acquiring meaningful capture data.
3. The repeat path alone is not sufficient, at least in the tested timing/state.
4. Interrupts may be optional, inactive, or require some OEM-specific arming sequence not yet reproduced.

## Immediate next steps suggested by this result

1. Repeat the post-init finger-hold capture to verify reproducibility.
2. Compare multiple non-zero captures for stability and spatial similarity.
3. Start adapting the EH577 libfprint draft around the post-init-led capture path.
4. Keep interrupt integration out of the first driver patch unless new evidence shows it is necessary.
