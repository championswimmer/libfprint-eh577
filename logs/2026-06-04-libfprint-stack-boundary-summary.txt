# EH577 libfprint stack boundary summary

Date: 2026-06-04
Context: after cold boot recovery, while debugging the EH577 runtime capture timeout.

## Live device state used for this check

- `lsusb` again shows `1c7a:0577`
- sysfs path: `/sys/bus/usb/devices/3-3`
- device node: `/dev/bus/usb/003/004`

## Tests run

### 1. Plain libusb smoketest

Command:

```bash
sudo -n ./build/eh577_libusb_smoketest --debug 4
```

Result:

- device opened successfully
- active configuration reported as `1`
- interface `0` claimed successfully
- first bulk OUT `45 47 49 53 60 00 fc` completed successfully
- `transferred=7`

Important meaning:

The first EH577 packet is writable through plain libusb with no timeout.

### 2. Plain libusb `set_configuration(1)` check

Command:

```bash
sudo -n ./build/eh577_libusb_smoketest --set-config --read --debug 4
```

Result:

- device opened successfully
- active configuration already reported as `1`
- `libusb_set_configuration(1)` returned `LIBUSB_ERROR_BUSY`

Important meaning:

Forcing configuration `1` on an already-configured EH577 is unnecessary and may be counterproductive.

### 3. Raw usbfs probe control

Command:

```bash
sudo -n ./build/eh577_usbfs_probe /dev/bus/usb/003/004 eh575-postinit 1
```

Result:

- TX `60 00 fc`
- write succeeded
- RX succeeded with `53 49 47 45 00 aa 01`

Important meaning:

The original direct usbfs path still works exactly as before on the cold-booted device.

### 4. Plain GUsb async smoketest

Command:

```bash
sudo -n ./build/eh577_gusb_smoketest --debug
```

Result:

- device found and opened successfully
- current configuration reported as `1`
- interface `0` claimed successfully
- async bulk OUT of `60 00 fc` completed successfully with `actual=7`

Important meaning:

The same GUsb async bulk primitive used by libfprint is capable of sending the first EH577 packet successfully outside the libfprint driver.

## Main conclusion

The old EH577 timeout is **not** explained by:

- hardware absence
- USB cable/power state after cold boot
- libusb bulk OUT behavior
- raw usbfs write behavior
- GUsb async bulk OUT behavior itself

So the remaining blocker is most likely in a higher layer, such as:

- the libfprint EH577 driver state machine / activation sequencing
- open-session behavior around the device before the first packet
- interaction between libfprint image-device activation and the EH577 request loop

## Immediate code change justified by this evidence

`g_usb_device_set_configuration(..., 1, ...)` was removed from `egis0577.c` again.

Reason:

- the device is already on configuration `1`
- plain libusb reports `set_configuration(1)` as `BUSY`
- plain GUsb async OUT works without any configuration reset
