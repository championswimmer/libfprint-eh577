# EH577 reset wedge / enumeration failure summary

Date: 2026-06-04
Context: runtime timeout investigation for the patched `egis0577` libfprint driver.

## What happened

A temporary `g_usb_device_reset()` experiment was inserted into `dev_init()` to see whether a reset before capture would unblock the first EH577 bulk OUT.

After that test, the EH577 disappeared from normal enumeration:

- `lsusb` no longer showed `1c7a:0577`
- `/sys/bus/usb/devices/3-3` disappeared
- direct probe commands could no longer open the old `/dev/bus/usb/...` node

## Kernel evidence

`journalctl -k` shows the EH577 port (`usb 3-3`) entering repeated recovery failures immediately after the reset attempt:

- `reset high-speed USB device number 4 using xhci_hcd`
- repeated `device descriptor read/64, error -110`
- repeated `device not accepting address ..., error -62`
- `usb usb3-port3: attempt power cycle`
- finally `usb usb3-port3: unable to enumerate USB device`

## Important conclusion

The runtime test with an explicit USB reset did not merely fail to move the capture timeout boundary; it pushed the sensor into a bad bus state where it stopped enumerating until physical recovery.

So for EH577 bringup:

- do **not** use `g_usb_device_reset()` in the normal libfprint open path
- treat USB reset as a high-risk operation on this hardware
- require a physical unplug/replug (or equivalent hardware recovery) before further live runtime testing once this state occurs

## Next step

After the device is physically present again, use the new `build/eh577_libusb_smoketest` helper to determine whether the first `60 00 fc` bulk OUT succeeds via plain libusb before changing more libfprint state-machine logic.
