# EH577 img-capture pre-init misbranch summary

Date: 2026-06-04
Command that successfully exposed the runtime behavior:

```bash
sudo -n sh -c 'LD_LIBRARY_PATH="$PWD/refs/libfprint/build/libfprint" G_MESSAGES_DEBUG=all "$PWD/refs/libfprint/build/examples/img-capture"'
```

## Observed runtime flow

The patched EH577 driver now runs far enough to exchange packets successfully.

Observed sequence:

1. device opens successfully
2. interface 0 is claimed successfully
3. libfprint activates the image device
4. post-init packet `60 00 fc` succeeds
   - RX: `53 49 47 45 00 aa 01`
5. post-init packet `60 01 fc` succeeds
   - RX: `53 49 47 45 01 01 01`
6. current driver logic treats that exact reply as a cue to switch into `PRE_INIT`
7. the PRE_INIT sequence then proceeds until packet `73 14 ec`
8. packet `73 14 ec` returns only a short 7-byte reply instead of the expected 5356-byte frame
9. libfprint aborts with:
   - `Unexpected short error of 7 size (expected 5356)`

## Why this matters

This matches the standalone probe evidence already collected earlier:

- the strongest meaningful-capture path on EH577 is the **post-init** path
- post-init packet `64 14 ec` has produced the expected 5356-byte payloads
- pre-init packet `73 14 ec` was already known to return only a short status reply in tested runs

So the previous EH575-style branch rule is actively harmful on EH577 in this runtime path.

## Code change made

The EH577 driver was patched to stay on the post-init path even when `60 01 fc` returns exact `SIGE 01 01 01`.

Files changed:

- `refs/libfprint/libfprint/drivers/egis0577.c`
- `refs/libfprint/libfprint/drivers/egis0577.h`
- synced into `wip-libfprint/drivers/egis0577.c`
- synced into `wip-libfprint/drivers/egis0577.h`
