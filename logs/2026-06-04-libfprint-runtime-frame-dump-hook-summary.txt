# EH577 libfprint runtime frame dump hook summary

Date: 2026-06-04
Context: after guided touch 04 confirmed that even forced post-init polling inside libfprint still produced only all-zero 5356-byte frames.

## Change made

Added an opt-in raw frame dump hook to the EH577 libfprint runtime path.

File changed:

- `refs/libfprint/libfprint/drivers/egis0577.c`
- synced to `wip-libfprint/drivers/egis0577.c`

## Behavior

If the environment variable below is set:

```bash
EGIS0577_FRAME_DUMP_DIR=/path/to/dump-dir
```

then every 5356-byte runtime frame seen by `save_img()` is dumped to that directory as:

- `0000-post-init-nonzero-0.bin`
- `0001-post-init-nonzero-0.bin`
- etc.

The dump includes the exact raw buffer received by libfprint at runtime.

## Why this matters

This enables direct comparison between:

- libfprint runtime-captured EH577 frames
- and the known-good standalone probe dumps from successful post-init finger-hold captures

That should tell us whether the runtime path is truly receiving different bytes, or whether the bytes are similar but later runtime logic is misclassifying them.

## Validation

Rebuilt and tested after adding the hook.
