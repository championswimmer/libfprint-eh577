## Summary

A second correctly timed guided post-init finger-hold run was captured to test reproducibility.

This second run again produced a meaningful non-zero `64 14 ec` payload, confirming that the earlier non-zero capture was not a one-off artifact.

## Interpretation

This strongly suggests the EH577 payload is capturing real, variable finger-dependent sensor data rather than a fixed template or deterministic noise block.

The current evidence now supports these claims with high confidence:

1. EH577 can produce meaningful image-like bulk data.
2. The successful path is post-init with finger contact timed appropriately.
3. The captured frame content varies between acquisitions, as expected for real scans.
4. The repeat path alone still does not appear sufficient in the tested conditions.

## Immediate implications for driver bringup

- The first EH577 libfprint driver should focus on the post-init-led bulk capture path.
- Interrupt support can remain optional/deferred unless later evidence shows it is required.
- Driver logic should avoid overfitting to single-state small reply bytes, but may use them as weak indicators.
