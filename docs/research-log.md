# EH577 Research Log Index

This repository contains the detailed chronological logs of investigations, experiments, and findings for the EgisTec EH577 fingerprint sensor. To keep logs manageable, the individual details are split into daily folders.

## [2026-06-12](research-log/2026-06-12)
* [01-enroll-wedge-root-cause.md](research-log/2026-06-12/01-enroll-wedge-root-cause.md): Traced the enroll wedge to an 8-large-read DMA ceiling per claim, identifying interface reset as the solution.
* [02-windows-inf-and-dll-guard-clues.md](research-log/2026-06-12/02-windows-inf-and-dll-guard-clues.md): Extracted registry INF keys and DLL function/variable strings pointing to detect/sensor splits.
* [03-sensor-size-research.md](research-log/2026-06-12/03-sensor-size-research.md): Compared EH577 raw/frame dimensions with other touch readers and concluded that `103x52` is likely the real raw frame while the current `208x104` snapshot output is not yet proven to be the best final matching geometry.
* [04-capture-noise-timing-and-denoise-findings.md](research-log/2026-06-12/04-capture-noise-timing-and-denoise-findings.md): Found same-finger score-0 is caused by fixed-pattern saturated hot-pixel noise (present in the no-finger baseline, rising with finger pressure); 3x3 median denoising removes it but erodes ridges, so the direction is fresh baseline subtraction + moderate-contact frame timing instead.
* [05-press-sensor-timing-strategies.md](research-log/2026-06-12/05-press-sensor-timing-strategies.md): Analyzed other libfprint PRESS type drivers to understand how they handle capture timing and finger settlement delays.
* [06-probe-vs-driver-init-diff.md](research-log/2026-06-12/06-probe-vs-driver-init-diff.md): Diffed probe vs driver init (PRE_INIT 29 / POST_INIT 18 byte-identical) and then empirically captured probe frames in all modes — the probe saturates today with/without pre-init and with/without USB reset, ruling out the init-diff/cadence/reset directions. Also dropped the old "clean reference" frame as pre-June-6 (swipe-era), so there is no validated evidence the sensor ever produced unsaturated frames.
* [07-windows-driver-capture-model-clues.md](research-log/2026-06-12/07-windows-driver-capture-model-clues.md): Searched the Windows EH577 package for capture-path clues; found a finger-down -> get-image -> quality/stability flow, no direct swipe/strip/stitch terminology, and Windows-side `52 x 103` raw-geometry corroboration.
* [08-enrollment-progress-api-and-ui-findings.md](research-log/2026-06-12/08-enrollment-progress-api-and-ui-findings.md): Checked libfprint, fprintd, and GNOME Settings behavior; confirmed the stack exposes enough information for exact enrollment progress (`completed / total`), and Ubuntu Settings not showing `11/15` appears to be a UI choice rather than a backend limitation.
* [09-ubuntu-gnome-fingerprint-ui-existing-issues.md](research-log/2026-06-12/09-ubuntu-gnome-fingerprint-ui-existing-issues.md): Identified the local OS as Ubuntu 25.10 / GNOME Settings 49.0, searched Ubuntu/GNOME issue trackers, and found no exact existing public bug for showing numeric fingerprint enrollment progress (`11/15`), though there is clear historical precedent for enrollment-stage visibility bugs and other current fingerprint UI issues.
* [10-small-sensor-partial-matcher-research.md](research-log/2026-06-12/10-small-sensor-partial-matcher-research.md): Researched how very small fingerprint sensors are handled in practice; found that multiple enrollment impressions, multiple stored partial templates or fused super-templates, quality filtering, and richer non-minutiae descriptors are the standard compensations, while libfprint's generic imaging path concretely does one-probe-vs-many-enrolled-partials matching.
* [11-stage1-capture12-quality-analysis.md](research-log/2026-06-12/11-stage1-capture12-quality-analysis.md): Analyzed the first 12-touch Stage-1 best-frame run; confirmed noisy captures still inflate minutiae, very clean captures can still be ridge-sparse, and the next gate needs both noise and ridge-content checks.
* [12-stage2-threshold-tuning-so-far.md](research-log/2026-06-12/12-stage2-threshold-tuning-so-far.md): Recorded the best live-tested Stage-2 operating thresholds so far (`grain<2.0%`, `minutiae>=2`, `ridge_pixels>=600`) and explicitly noted that the low-minutiae problem remains unsolved.
* [13-image-enhancement-strategy-for-low-minutiae.md](research-log/2026-06-12/13-image-enhancement-strategy-for-low-minutiae.md): Researched fingerprint enhancement approaches and ran offline PGM transforms; after visual review, selected `stretch5` as the current driver-side enhancement while retaining cross-press Bozorth validation as the real success criterion.
* [14-stretch5-noise-separation-and-p5-gate.md](research-log/2026-06-12/14-stretch5-noise-separation-and-p5-gate.md): Analyzed the first stretch5 live run; found stretch5 improves visibility but can hide noise separation, so added a pre-stretch p5 gate, later relaxed it to `p5>=100`, and relaxed enhanced-grain default to `8.0%`.
* [15-retry-wedge-after-minutiae-gate.md](research-log/2026-06-12/15-retry-wedge-after-minutiae-gate.md): Diagnosed a capture retry wedge after raising the minutiae floor; the helper now asks for lift and waits before retrying so the driver can observe a clean no-finger baseline.
* [16-capture12-retry-startup-transient-wedge.md](research-log/2026-06-12/16-capture12-retry-startup-transient-wedge.md): Diagnosed a later capture12 hang as the driver looping on persistent finger-like startup transients, not log blocking; added a 750ms force-arm cap and made capture12 debug quiet by default.
* [17-driver-noisy-burst-recovery.md](research-log/2026-06-12/17-driver-noisy-burst-recovery.md): Implemented driver-side, action-aware noisy-burst recovery: repeated noise-like rejects now clear the warm baseline, force a fresh pre-init/reinit cycle, and require clean baseline frames before arming again.
